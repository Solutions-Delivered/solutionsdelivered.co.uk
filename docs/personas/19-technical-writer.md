# Persona: Technical Writer

## Profile
- **Name:** Michael Chen
- **Age:** 40
- **Role:** Senior Technical Writer
- **Company Type:** Software Company
- **Experience:** 15 years in technical documentation
- **Location:** Edinburgh, UK

## Background
Michael creates clear, accurate documentation for technical products and services. He specializes in translating complex technical concepts into understandable language for different audiences. He values clarity, accuracy, and usability in documentation.

## Goals & Motivations
- Create documentation that helps users succeed
- Make complex concepts understandable
- Reduce support burden through better documentation
- Organize information logically and accessibly
- Keep documentation accurate and up-to-date
- Write for different technical proficiency levels

## Pain Points & Challenges
- Subject matter experts who struggle to explain concepts
- Rapidly changing products that make docs obsolete
- Limited time to understand complex technical topics
- Users with varying technical backgrounds
- Lack of feedback on documentation effectiveness
- Documentation that's hard to maintain or search
- Balancing comprehensiveness with conciseness

## What They Look For in a Website
- **Clear Structure:** Logical organization of documentation
- **Search Functionality:** Robust search to find information quickly
- **Versioning:** Documentation for different product versions
- **Code Examples:** Practical, working examples
- **Visual Aids:** Diagrams, screenshots, videos where helpful
- **Consistent Formatting:** Predictable structure and style
- **Progressive Disclosure:** Basic info upfront, details available
- **Contextual Help:** Documentation linked from relevant places in product
- **Accuracy:** Technically correct and regularly updated
- **Glossary:** Definitions of technical terms
- **Getting Started Guides:** Clear onboarding documentation
- **Troubleshooting:** Common problems and solutions

## How They Evaluate Personas
- **Technical Proficiency:** Beginner, intermediate, advanced, or expert?
- **Documentation Needs:** Tutorials, how-tos, reference, troubleshooting?
- **Learning Style:** Step-by-step, conceptual, example-driven?
- **Use Cases:** What tasks are they trying to complete?
- **Prior Knowledge:** What can we assume they already know?
- **Context of Use:** When and where do they access documentation?
- **Time Constraints:** Do they need quick answers or deep understanding?
- **Technical Vocabulary:** What terms do they understand?
- **Documentation Format:** Text, video, interactive, or mixed?

## Technical Proficiency
- Expert: Technical writing, documentation standards, information design
- Advanced: API documentation, developer documentation, diagramming tools
- Intermediate: HTML/CSS, Markdown, version control, basic coding
- Tools: MadCap Flare, Adobe FrameMaker, Confluence, Markdown editors, Swagger

## Preferred Communication Channels
- Documentation style guides
- Information architecture diagrams
- Documentation templates and standards
- Technical specifications
- User feedback on documentation

## Key Quotes
> "Good documentation should make the complex simple, not the simple complex."

> "If users are calling support, the documentation has failed."

> "Documentation is a product feature, not an afterthought."

## Red Flags for This Persona
- No clear documentation structure or navigation
- Missing search functionality
- Outdated or inaccurate information
- Jargon without explanation
- No code examples or visual aids
- Inconsistent formatting or style
- Documentation that assumes too much knowledge
- No troubleshooting or FAQ section
- Poor mobile experience for docs
- No way to provide feedback on docs

## Website Evaluation Checklist
- [ ] Is documentation clearly organized?
- [ ] Is there robust search functionality?
- [ ] Is information accurate and current?
- [ ] Are technical terms defined?
- [ ] Are there practical code examples?
- [ ] Are visual aids used appropriately?
- [ ] Is formatting consistent?
- [ ] Are there getting started guides?
- [ ] Is there progressive disclosure of detail?
- [ ] Can users provide feedback on docs?
- [ ] Are common problems addressed?
- [ ] Is documentation versioned if needed?
- [ ] Does it work well on mobile?
- [ ] Is language clear and concise?
- [ ] Are different skill levels accommodated?
