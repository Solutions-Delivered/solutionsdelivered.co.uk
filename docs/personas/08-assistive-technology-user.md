# Persona: Assistive Technology User

## Profile
- **Name:** David Williams
- **Age:** 52
- **Role:** Senior Business Analyst
- **Disability:** Blind since age 23 (degenerative condition)
- **Experience:** 30+ years using screen readers professionally
- **Location:** Leeds, UK

## Background
David is a power user of assistive technology, primarily using JAWS screen reader on Windows at work and VoiceOver on his iPhone. He's been navigating the web since the early days and has seen both improvements and ongoing frustrations with web accessibility. He's efficient with keyboard shortcuts and has little patience for poorly coded websites.

## Goals & Motivations
- Complete tasks efficiently without encountering barriers
- Access information independently without asking for help
- Use the same websites and services as everyone else
- Advocate for better accessibility in his workplace
- Stay current with technology and professional development

## Pain Points & Challenges
- Websites that don't work with screen readers
- Unlabeled buttons and form fields
- Complex navigation structures
- Images without alt text, especially in critical content
- Pop-ups and modals that trap keyboard focus
- Content that updates without announcement
- CAPTCHAs without audio alternatives
- PDF documents that aren't screen reader accessible
- Automatic carousels that interfere with reading
- Poorly structured tables

## What They Look For in a Website
- **Semantic HTML:** Proper use of headings, landmarks, lists, tables
- **Keyboard Accessibility:** Everything works without a mouse
- **Descriptive Labels:** Form fields, buttons, and links clearly described
- **Alt Text:** Meaningful descriptions for images (not "image123.jpg")
- **Skip Links:** Ability to skip repetitive navigation
- **ARIA Landmarks:** Main, navigation, complementary regions properly marked
- **Focus Management:** Logical tab order, visible focus indicators
- **Announcements:** Screen reader notifications for dynamic content changes
- **Consistent Navigation:** Predictable structure across pages
- **Accessible Forms:** Proper field labels, error identification, instructions
- **Table Headers:** Proper header associations for data tables
- **Link Context:** Clear link purpose from text alone

## How They Evaluate Personas
- **Actual Experience:** Do personas reflect real assistive technology users?
- **Specific Tools:** Which screen readers, magnifiers, or input devices do they use?
- **Proficiency Level:** Are they power users or beginners?
- **Context:** Where and how do they access websites (work, home, mobile)?
- **Barriers:** What specific issues do they commonly encounter?
- **Workarounds:** What strategies do they use to overcome barriers?
- **Frustration Points:** What causes them to abandon sites?
- **Success Factors:** What makes a website work well for them?
- **Authenticity:** Were real users with disabilities consulted?

## Technical Proficiency
- Expert: JAWS, VoiceOver, keyboard navigation, accessibility features
- Advanced: Windows accessibility features, iOS accessibility
- Intermediate: Basic HTML understanding, web navigation strategies
- Tools: JAWS (primary), VoiceOver (iPhone), keyboard shortcuts

## Preferred Communication Channels
- Phone calls (prefers voice over text in many contexts)
- Email with clear subject lines and structure
- Accessible documents (properly tagged PDFs or HTML)
- Audio descriptions for video content
- Text transcripts for audio content

## Key Quotes
> "If I can't tab to it, it doesn't exist for me."

> "Alt text that says 'image' is worse than no alt text—it wastes my time."

> "I don't need a 'screen reader mode.' I need you to build it right the first time."

## Red Flags for This Persona
- "Click here" or "Read more" links without context
- Images with no alt text or generic descriptions
- Forms with placeholder text but no labels
- Custom widgets that don't work with keyboard
- Content that appears visually but not to screen readers
- Modals that trap focus or can't be closed
- Auto-playing videos or carousels
- Tables without proper header associations
- Navigation that changes structure between pages

## Website Evaluation Checklist
- [ ] Can I navigate the entire site using only keyboard?
- [ ] Do all images have meaningful alt text?
- [ ] Are all form fields properly labeled?
- [ ] Is there a skip navigation link?
- [ ] Do headings create a logical outline?
- [ ] Are ARIA landmarks used appropriately?
- [ ] Do dynamic updates announce to screen readers?
- [ ] Are error messages clear and associated with fields?
- [ ] Can I understand link purpose from the link text alone?
- [ ] Do custom widgets announce their role and state?
- [ ] Are tables properly structured with headers?
- [ ] Can I complete all tasks independently?
